## Constraints / Risk Factors

* High overload risk from giant roadmaps.
* Low tolerance for ambiguous tasks.
* Tooling friction around Git, Docker, terminal, and environment issues.
* Career/job-market discussion can derail technical learning.
* Needs one active module at a time.
* Needs evidence-based evaluation, not assumed ability.
* Needs small tasks with explicit pass/fail criteria.
* Should avoid new tools before current module is evaluated.
* Should avoid combining technical diagnostics with career panic in the same session.
* Should avoid treating project existence as proof of independent competence.
* Should distinguish guided implementation from independent reproduction.

## Learning Design Requirements

The education system should use:

* external storage as source of truth
* one active module at a time
* small diagnostic tasks
* pass/fail rubrics
* explicit remediation steps
* regular but limited market probes
* profile updates only after evidence
* separate files for profile, target, module, market, and evaluation logs
